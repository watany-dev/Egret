# update-design

設計書の品質を評価し、改善提案を行うスキル。

## Phase 1: 設計書の収集

1. `docs/design/` ディレクトリ配下のすべての `.md` ファイルを読み込む
2. 各設計書のタイトル・構成を把握する
3. 設計書が存在しない場合は、その旨を報告して終了する

## Phase 2: 設計書の評価

各設計書を以下のカテゴリで評価する:

### 評価カテゴリ

1. **モジュール・構造体設計**
   - Rustモジュールの責務分離が適切か
   - 構造体のフィールド定義が明確か
   - トレイト実装が適切に設計されているか

2. **Docker API (bollard) / ECS互換設計**
   - bollard クレートとの連携設計が適切か
   - ECSメタデータエンドポイントの互換性が考慮されているか
   - クレデンシャルプロバイダの設計が適切か

3. **エラーハンドリング設計**
   - `anyhow`/`thiserror` の使い分けが適切か
   - エラー型の階層構造が明確か

4. **技術選定の根拠**
   - clap, bollard, tokio, axum 等の選定理由が明記されているか
   - 代替案との比較が記載されているか

5. **データフロー**
   - 入力（タスク定義JSON）から出力（コンテナ実行）までのフローが明確か
   - dependsOn DAG の処理フローが記載されているか

### 評価基準

各カテゴリを以下の基準で評価:
- ✅ **十分**: 設計が明確で実装に移れる状態
- ⚠️ **改善推奨**: 理解可能だが補足が望ましい
- ❌ **要改善**: 設計が不明確で実装前に修正が必要

## Phase 3: ソースコードとの整合性チェック

1. `src/` 配下のRustソースコードを参照する
2. 設計書に記載されたモジュール・構造体・関数シグネチャが実装と一致しているか確認する
3. 設計書に記載されているが未実装の機能を特定する
4. 実装されているが設計書に記載されていない機能を特定する

## Phase 4: 改善提案の出力

以下の形式で改善提案を出力する:

```markdown
## 設計書評価レポート

### [設計書名]

#### 評価サマリ
| カテゴリ | 評価 |
|---------|------|
| モジュール・構造体設計 | ✅/⚠️/❌ |
| Docker API / ECS互換設計 | ✅/⚠️/❌ |
| ... | ... |

#### 改善提案
1. [具体的な改善内容]
2. [具体的な改善内容]

#### ソースコードとの差分
- [差分の詳細]
```

## 記述ルール

- コード例はRustで記述すること
- 設計書は日本語で記述すること
- 構造体名・関数名はRustの命名規則（snake_case / PascalCase）に従うこと
