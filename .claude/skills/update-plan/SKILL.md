# update-plan

プランモードで実装計画を完成させた直後、ユーザーに提示する直前に発動する統合検証・改善スキル。
update-design の全機能（設計書品質評価）を内包しつつ、ロードマップ・要件定義・ソースコードとの横断的整合性チェックと、プラン自体の品質向上を行う。

## 発動タイミング

**プランモードでプラン完了と判断した直後、`ExitPlanMode` を呼ぶ直前**に本スキルを実行する。

フロー:
1. ユーザーが実装タスクを依頼する
2. Claude がプランモードで調査・計画を作成する
3. プランが完成した時点で **本スキルを発動**
4. 本スキルの検証結果に基づきプランを改善する
5. 改善済みプランで `ExitPlanMode` を呼ぶ

## Phase 1: コンテキスト収集

### 1-1. プランの読み込み

1. 作成中のプランファイルの内容を把握する
2. プランが対象とする機能・Phase・モジュールを特定する

### 1-2. 関連ドキュメントの収集

1. `docs/design/` 配下のすべての `.md` ファイルを読み込む
2. `docs/ROADMAP.md` を読み込む
3. `docs/requirements.md` を読み込む
4. `CLAUDE.md` を読み込む（プロジェクト方針の確認）

### 1-3. ソースコード収集

1. プランが対象とするモジュールの `src/` 配下のソースコードを読み込む
   - `src/main.rs` — エントリポイント
   - `src/cli/` — CLIコマンド定義
   - `src/taskdef/` — タスク定義パーサ・型
   - `src/container/` — OCI コンテナランタイムクライアント
   - `src/orchestrator/` — コンテナライフサイクル管理
   - `src/metadata/` — メタデータエンドポイント
   - `src/credentials/` — クレデンシャルプロバイダ
   - `src/secrets/` — シークレットリゾルバ
   - `src/overrides/` — オーバーライド設定
2. `Cargo.toml` の依存クレート一覧を確認する
3. 公開API・構造体・トレイトの一覧を把握する
4. テストコード（`#[cfg(test)]` モジュール、`tests/` ディレクトリ）を把握する

## Phase 2: 設計書品質評価（update-design 完全互換）

プランが関連する設計書を以下のカテゴリで評価する。

### 評価カテゴリ

1. **モジュール・構造体設計**
   - Rustモジュールの責務分離が適切か
   - 構造体のフィールド定義が明確か
   - トレイト実装が適切に設計されているか

2. **Docker API (bollard) / ECS互換設計**
   - bollard クレートとの連携設計が適切か
   - ECSメタデータエンドポイントの互換性が考慮されているか
   - クレデンシャルプロバイダの設計が適切か

3. **エラーハンドリング設計**
   - `anyhow`/`thiserror` の使い分けが適切か
   - エラー型の階層構造が明確か

4. **技術選定の根拠**
   - clap, bollard, tokio, axum 等の選定理由が明記されているか
   - 代替案との比較が記載されているか

5. **データフロー**
   - 入力（タスク定義JSON）から出力（コンテナ実行）までのフローが明確か
   - dependsOn DAG の処理フローが記載されているか

### 評価基準

各カテゴリを以下の基準で評価:
- ✅ **十分**: 設計が明確で実装に移れる状態
- ⚠️ **改善推奨**: 理解可能だが補足が望ましい
- ❌ **要改善**: 設計が不明確で実装前に修正が必要

## Phase 3: ソースコードとの深層整合性チェック

update-design の整合性チェックを拡張し、より深い分析を行う。

### 3-1. シグネチャ整合性（update-design 互換）

1. 設計書に記載されたモジュール・構造体・関数シグネチャが実装と一致しているか確認する
2. 設計書に記載されているが未実装の機能を特定する
3. 実装されているが設計書に記載されていない機能を特定する

### 3-2. テストカバレッジギャップ分析

1. 設計書の「テスト戦略」セクションに記載されたテストケースが実装されているか確認する
2. 実装済みテストが設計書に記載されていないケースをカバーしているか確認する
3. CLAUDE.md の 95% カバレッジ要件に対する懸念箇所を特定する

### 3-3. エラーハンドリングパターン分析

1. 設計書のエラー型定義と実装のエラー型を比較する
2. `unwrap` / `expect` の使用がないか確認する（CLAUDE.md で禁止）
3. `anyhow` と `thiserror` の使い分けが設計方針と一致しているか確認する

### 3-4. 依存クレート整合性分析

1. `Cargo.toml` の `[dependencies]` と設計書の「技術選定」セクションを比較する
2. 設計書に記載されているが `Cargo.toml` にないクレートを特定する
3. `Cargo.toml` にあるが設計書で言及されていないクレートを特定する
4. ロードマップの Phase 列と実際の導入タイミングの整合性を確認する

## Phase 4: ドキュメント横断整合性チェック

ロードマップ・要件定義・設計書の三者間の整合性を検証する。

### 4-1. ロードマップ ↔ 設計書

1. プランが対象とする Phase に対応する設計書が存在するか確認する
2. 設計書が存在する Phase のステータスが適切か確認する
3. 設計書の内容がロードマップの目標・タスクリストと一致しているか確認する

### 4-2. ロードマップ ↔ 要件定義

1. プランが対象とする Phase が要件定義の FR/NFR のどれに対応するか確認する
2. 要件定義の状態（✅ 実装済み / 🔲 未実装）がロードマップのステータスと一致しているか確認する
3. ロードマップにあるが要件定義にない機能を特定する
4. 要件定義にあるがロードマップにない要件を特定する

### 4-3. 要件定義 ↔ ソースコード

1. 「✅ 実装済み」の要件が実際にソースコードに実装されているか確認する
2. 「🔲 未実装」の要件に関連するコードが存在しないか確認する（先行実装の検出）

### 4-4. ロードマップ ↔ ソースコード

1. ロードマップで「✅ 完了」の Phase の全タスクが実装されているか確認する
2. ロードマップで未完了の Phase に関連する先行実装がないか確認する

## Phase 5: プラン品質検証

作成中のプラン自体を検証し、品質を高める。

### 5-1. CLAUDE.md 方針との整合性

1. **TDDサイクル**: プランがRed→Green→Refactorの順序で記述されているか
2. **Tidy First?**: 構造的変更と機能的変更が分離されているか
3. **イテレーション単位**: 各ステップが最小単位で分割されているか
4. **完了要件**: `make check` の全ステップ（fmt-check, lint, test, doc, deny）が考慮されているか

### 5-2. 設計書フォーマットとの整合性

プランが新しい設計書の作成を含む場合、既存設計書の標準セクション構成に従っているか確認する:

```markdown
# [機能名] 設計書

## 概要
## 設計方針
## 技術選定
## 型定義
## エラー型
## 公開API
## データフロー
## テスト戦略
## イテレーション
```

### 5-3. 未着手フェーズの設計書提案

プランが対象とする Phase に設計書が存在しない場合:

1. 上記の標準セクション構成に従った設計書アウトラインをプランに追加する
2. ロードマップの目標・タスクリストから主要な型定義・公開APIを推定する
3. 依存関係グラフに基づく実装順序をイテレーションセクションとして提案する

### 5-4. 影響範囲の網羅性

1. プランが変更するモジュールに依存する他のモジュールへの影響を確認する
2. プランに含まれていない必要な変更（テスト追加、ドキュメント更新等）を特定する
3. ロードマップ・要件定義のステータス更新が必要か確認する

## Phase 6: プラン改善とレポート

### 6-1. プランへのフィードバック反映

Phase 2〜5 で発見した問題をプランに反映する:

1. **P0（必須修正）**: プランの実行を阻害する不整合・欠落の修正
   - 設計書との矛盾
   - 要件定義に存在しない機能の追加
   - CLAUDE.md 方針への違反
2. **P1（推奨修正）**: プランの品質を向上させる改善
   - テスト戦略の補完
   - イテレーション分割の改善
   - 影響範囲の追加記載
3. **P2（情報提供）**: 参考情報として記載
   - 既存設計書の改善余地
   - 将来Phaseへの影響

プランファイルの末尾に以下の検証サマリを追記する:

### 6-3. 完了アクション

1. P0 の修正をプランに反映する
2. P1 の修正をプランに反映する（可能な範囲で）
3. 検証サマリを付加した状態でプランをユーザーに提示する（`ExitPlanMode` を実行する）

## 記述ルール

- コード例はRustで記述すること
- 設計書・レポートは日本語で記述すること
- 構造体名・関数名はRustの命名規則（snake_case / PascalCase）に従うこと
- 設計書のセクション構成は既存設計書（概要→技術選定→型定義→エラー型→公開API→データフロー→テスト戦略→イテレーション）に従うこと
- ロードマップ・要件定義のステータスは実装の実態に基づき、推測で変更しないこと
- プランの修正は根拠（設計書・要件定義・CLAUDE.md の該当箇所）を明示すること
