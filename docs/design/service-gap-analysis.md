# ECS Service 対応 — ギャップ分析

## Context

Egret は Phase 0-6 で ECS タスクランナーとしての主要機能を実装済みである。本ドキュメントは「ECS Service 対応」に何が足りないかを分析し、実装パスを整理する。

ECS の Service は「タスク定義を継続的に実行し、desired count を維持する」制御プレーンであり、タスクランナーとは根本的に異なるライフサイクルモデルを持つ。Egret の設計理念「ECS control plane の再現ではなく、ECS アプリが期待する実行時契約をローカルで満たす」に沿い、ローカル開発で必要な範囲に絞ってギャップを分析する。

---

## タスク vs サービス: ライフサイクルの比較

```
【タスク (現在の Egret)】

  egret run → DAG解決 → コンテナ起動 → 完了待ち → 終了・クリーンアップ
              ^^^^^^^^   ^^^^^^^^^^^^   ^^^^^^^^   ^^^^^^^^^^^^^^^^^^
              一度きり    一度きり        一方向      自動終了

【サービス (ECS Service)】

  create-service → 初回デプロイ ──────────────────────────────────┐
                                                                  │
       ┌──────────────────── reconciliation loop ◄────────────────┘
       │
       ▼
  desired_count と actual count を比較
       │
       ├─ 不足 → 新タスク起動 → ヘルスチェック → running に加算
       ├─ 過剰 → タスク停止（draining）→ running から減算
       ├─ 異常 → 失敗タスク置換 → リスタートポリシー適用
       └─ 正常 → 待機（次のイベントまで）
       │
       ├─ update-service → ローリングアップデート開始
       └─ delete-service → 全タスク停止 → クリーンアップ
```

## 具体的な実装ギャップ

### Gap 1: リスタートポリシー

| 項目 | 内容 |
|------|------|
| **重要度** | 最高（サービスの最小要件） |
| **現状** | essential コンテナ終了 → タスク全体停止 (`src/orchestrator/mod.rs`) |
| **必要** | コンテナ障害時の自動再起動 |
| **影響モジュール** | `orchestrator`, `taskdef`, `cli` |

**詳細**:
- ECS Service は `restartPolicy` で `NONE` / `ON_FAILURE` / `ALWAYS` を指定
- `ON_FAILURE`: 非ゼロ exit code の場合のみ再起動
- `ALWAYS`: exit code に関係なく再起動
- 指数バックオフ（1s → 2s → 4s → ... → 最大 300s）
- 最大リスタート回数の制御（ECS は無制限だがローカルでは制限が実用的）

**現在のコード** (`src/orchestrator/mod.rs`):
- `wait_for_condition()` でコンテナの状態を監視
- essential コンテナ終了時に即座にタスク全体を停止
- ここにリスタート判定ロジックを挿入する必要がある

---

### Gap 2: 継続的 Reconciliation ループ

| 項目 | 内容 |
|------|------|
| **重要度** | 高（サービスモードの中核） |
| **現状** | ワンショット: 起動 → 完了待ち → 終了 |
| **必要** | desired state と actual state の継続的な比較・収束 |
| **影響モジュール** | `orchestrator`（大規模拡張） |

**詳細**:
- 現在の orchestrator は「一方向フロー」: DAG 解決 → 順次起動 → 完了待ち
- サービスモードでは「イベントループ」が必要:
  1. コンテナ状態の定期ポーリング（`inspect_container`）
  2. 期待状態との差分検出
  3. 収束アクション実行（起動/停止/再起動）
  4. 1 に戻る
- tokio の `select!` + `interval` で実装可能
- 既存の `ContainerRuntime` トレイトの操作は十分（追加メソッド不要）

---

### Gap 3: サービス定義スキーマ

| 項目 | 内容 |
|------|------|
| **重要度** | 高（設定の入り口） |
| **現状** | `TaskDefinition` のみ (`src/taskdef/mod.rs`) |
| **必要** | サービスレベルの設定フィールド |
| **影響モジュール** | `taskdef` |

**追加が必要なフィールド**:
```json
{
  "serviceName": "my-service",
  "desiredCount": 1,
  "deploymentConfiguration": {
    "maximumPercent": 200,
    "minimumHealthyPercent": 100
  },
  "restartPolicy": {
    "enabled": true,
    "restartCondition": "ON_FAILURE",
    "maxRestarts": 10,
    "backoffSeconds": 300
  }
}
```

**注意**: ECS では Service 定義と Task 定義は別リソース。Egret でのアプローチ:
- **案A**: 別ファイル（`egret-service.json`）で Service 設定を定義
- **案B**: CLI フラグ（`--service --desired-count 1 --restart on-failure`）
- **案C**: タスク定義に Egret 独自フィールドを拡張（`x-egret-service` セクション）

---

### Gap 4: CLI コマンド

| 項目 | 内容 |
|------|------|
| **重要度** | 高（ユーザーインターフェース） |
| **現状** | `egret run`（ワンショット実行） |
| **必要** | サービスモードの起動・管理 |
| **影響モジュール** | `cli/run.rs` または新規 `cli/service.rs` |

**選択肢**:
- **案A**: `egret run --service` フラグ追加（最小変更）
- **案B**: `egret service {start|stop|scale|update}` サブコマンド（拡張性重視）

---

### Gap 5: ローリングアップデート

| 項目 | 内容 |
|------|------|
| **重要度** | 中（MVP では不要） |
| **現状** | `egret stop` → `egret run`（手動停止→再起動） |
| **必要** | 無停止でのタスク定義更新 |
| **影響モジュール** | `orchestrator`（新規ロジック） |

**詳細**:
- `maximumPercent: 200` → 新旧合わせて 200% まで許容（新コンテナ先に起動）
- `minimumHealthyPercent: 100` → 常に desired_count 以上の healthy コンテナを維持
- フロー: 新コンテナ起動 → ヘルスチェック通過 → 旧コンテナ停止
- ローカル開発では single replica が主流のため、MVP では停止→起動で十分

---

### Gap 6: クレデンシャルローテーション

| 項目 | 内容 |
|------|------|
| **重要度** | 中（長時間稼働で必要） |
| **現状** | 起動時に1回取得、12時間固定 TTL (`src/credentials/mod.rs`) |
| **必要** | バックグラウンドでの定期リフレッシュ |
| **影響モジュール** | `credentials` |

**詳細**:
- 現在 `load_credentials()` は起動時に1回だけ呼ばれる
- STS 一時クレデンシャルは通常 1-12 時間で期限切れ
- サービスモードでは TTL の半分の間隔でリフレッシュが必要
- `tokio::spawn` でバックグラウンドタスクとして実装可能
- メタデータサーバー (`src/metadata/mod.rs`) の `AppState` に `RwLock<Credentials>` を持たせる

---

### Gap 7: ロードバランサー / サービスディスカバリ

| 項目 | 内容 |
|------|------|
| **重要度** | 低（ローカル開発では通常不要） |
| **現状** | bridge ネットワーク + コンテナ名 DNS エイリアス |
| **必要** | 複数レプリカ間のトラフィック分散 |
| **影響モジュール** | 新規モジュール（`src/loadbalancer/`） |

**詳細**:
- 複数レプリカを起動した場合、外部からのアクセスをどのコンテナに振るか
- 既存の axum サーバーを拡張してリバースプロキシを実装する案
- Docker の DNS ラウンドロビン（同一ネットワークエイリアス）で簡易対応する案
- ローカル開発では desired_count=1 が主流のため、MVP では不要

---

### Gap 8: メタデータエンドポイントのサービス情報

| 項目 | 内容 |
|------|------|
| **重要度** | 低 |
| **現状** | タスクレベルのメタデータのみ (`src/metadata/mod.rs`) |
| **必要** | サービス名・デプロイ状態などのメタデータ |
| **影響モジュール** | `metadata` |

**詳細**:
- ECS V4 メタデータにはサービス名は含まれない（タスクメタデータのみ）
- ただし `ECS_SERVICE_NAME` 環境変数をアプリが参照するケースがある
- メタデータ拡張よりも環境変数注入で対応可能

## 実装パスの提案

### MVP (最小実用パス)

ローカル開発の「サービスっぽく動かしたい」ニーズに最小限で応える:

| # | 機能 | Gap | 工数感 |
|---|------|-----|--------|
| 1 | `--service` フラグ | Gap 4 | 小 |
| 2 | リスタートポリシー | Gap 1 | 中 |
| 3 | クレデンシャルリフレッシュ | Gap 6 | 小 |

**MVP で得られるもの**:
- コンテナが crash しても自動で再起動される
- `egret run --service` でタスクが「サービスのように」常時稼働する
- 長時間稼働でクレデンシャルが切れない

### Phase 2 (実用拡張)

| # | 機能 | Gap | 工数感 |
|---|------|-----|--------|
| 4 | サービス定義スキーマ | Gap 3 | 中 |
| 5 | Reconciliation ループ | Gap 2 | 大 |
| 6 | ローリングアップデート | Gap 5 | 大 |

### Phase 3 (フル機能)

| # | 機能 | Gap | 工数感 |
|---|------|-----|--------|
| 7 | ロードバランサー | Gap 7 | 大 |
| 8 | サービスメタデータ | Gap 8 | 小 |

## 既存コードの再利用可能性

現在のコードベースはサービス対応に向けて良い土台がある:

| 既存資産 | サービスでの活用 |
|---------|----------------|
| `ContainerRuntime` トレイト | そのまま再利用可能。追加メソッド不要 |
| `Orchestrator::start()` | 初回起動ロジックはそのまま使える |
| `wait_for_condition()` | ヘルスチェックポーリングを reconciliation に転用 |
| メタデータサーバー (axum) | リバースプロキシの基盤として拡張可能 |
| `ContainerConfig` | レプリカ生成時のテンプレートとして使える |
| bridge ネットワーク | 複数レプリカを同一ネットワークに配置可能 |

---

## まとめ

Egret が ECS Service に対応するために最も不足しているのは:

1. **リスタートポリシー** — コンテナ障害からの自動復旧
2. **サービスモード** — ワンショットではなく常時稼働するライフサイクル
3. **Reconciliation ループ** — desired state を維持する制御ループ

これらは既存のタスクランナーアーキテクチャの延長線上にあり、`ContainerRuntime` トレイトや orchestrator の基盤は再利用できる。ただし orchestrator は「一方向フロー → イベントループ」への転換が必要であり、これが最大の構造的変更となる。
